# Решение проблем

## Нет интернета / "Could not fetch"

1. Проверьте, что Switch подключен к Wi-Fi и страница
   <https://api.github.com> открывается во встроенном web-браузере.
2. Откройте `Tools → Проверка сети` — она покажет статус всех
   зависимых хостов (api.github.com, raw.githubusercontent.com,
   Telegram, репо пака, апстрим). Если все красные — проблема в DNS
   или провайдере.
3. Попробуйте сменить DNS в `Tools → Редактировать настройки
   интернета` — мы добавили предустановки Cloudflare, Google, Quad9,
   Яндекс.DNS, AdGuard.
4. Если только GitHub красный — это типичный rate-limit. Через час
   само пройдёт; нативно лимит снимается аутентификацией, которой у
   нас нет.

## GitHub API возвращает 403

GitHub лимитирует анонимные API-запросы: 60/час на IP. Если в течение
часа вы много раз нажимали "Обновить" — словите 403. Подождите час
или подключитесь к другой сети. Версии тулзов после этого
автоматически восстановятся при следующем запуске.

## NACP не парсится / "not installed or unreadable NACP"

Это нормальное состояние, если соответствующий .nro не установлен
(например `FPSLocker` нет на SD — version_check просто отметит как
"not installed"). Если же файл есть, но всё равно "unreadable":

- Проверьте, что файл не повреждён: размер > 0, открывается hbmenu.
- Поддерживаются только .nro в формате NRO+ASET (стандартный для
  homebrew). Старые `<2019` .nro без ASET-секции — не парсятся.
- Запустите `Tools → Создать диагностический отчёт` и приложите
  получившийся `diag-*.txt` к issue.

## Sigpatches outdated после обновления HOS

После каждого major-обновления Switch системы sigpatches нужно
переустанавливать заново. Самый простой способ:

1. `Tools → Установить пак Ряженки` — заберёт актуальный пак целиком,
   включая sigpatches. Опция `auto_backup_before_pack` (включена по
   умолчанию) предварительно бэкапит `/atmosphere/` в `/aio-backup/`.
2. Альтернатива: только sigpatches. Скоро в Tools, пока — через
   "Custom Downloads" с прямой ссылкой.

## Пак не распаковывается / "extract error"

1. Свободное место. Pre-flight требует 500 MiB; пак Ряженки
   распаковывается на 1+ GiB. Освободите место (можно через
   `Tools → Очистить старые резервные копии`).
2. FAT32 vs exFAT. exFAT под Switch официально supported, но если
   карта была повреждена — рекомендуем переформатировать в FAT32
   (стандартный совет community).
3. Сам архив. Откройте `Tools → Проверка сети` и убедитесь, что
   `github.com/Dimasick-git/Ryzhenka` доступен. Если архив скачался
   битый — `Tools → Удалить скачанные файлы` и повторите.

## Приложение запустилось не на русском

При первом запуске форк проверяет
`/config/aio-switch-updater/language.json`. Если файла нет — ставится
русский. Если файл есть, но указан другой язык — открыть
`Tools → Изменить язык приложения` и выбрать русский, либо удалить
`language.json` и перезапустить.

## Самообновление не работает

Самообновление качает
`releases/latest/download/Ryazhenka_AIO.zip` — этот файл всегда есть,
потому что CI делает release при каждом push в `main` с
`make_latest=true`. Если "Update available" не появляется:

1. Проверьте `Tools → Проверка сети` — `api.github.com` должен быть
   OK.
2. Проверьте версию текущего .nro: должна совпадать с тегом из
   `Releases`. Иначе кнопка "Обновить" появится в верхней части
   Tools.
3. Откройте лог приложения (`Tools → Показать лог приложения`) — там
   ищите строки `version_check:` и `compareVersions`.

## Краш при старте

Crash handler (`ryazhenka_crash_handler.cpp`) пишет последнее
исключение в `/config/aio-switch-updater/log.txt` перед `abort`.
Если файла лога нет — включите `"verbose_log": true` в
`/config/aio-switch-updater/config.json` и попробуйте снова.

Полезные команды для bug-report:
- `Tools → Создать диагностический отчёт` — собирает sysinfo +
  version_check + последние 200 строк лога в один `diag-*.txt` под
  `/switch/aio-switch-updater/`.
- Приложите этот файл к issue.

## График CPU/Skin температуры плоский на нуле

Сервис `ts` (`tsInitialize`) не инициализировался. Проверьте в
`Tools → Показать лог приложения`:

```
[WARN] metrics: tsInitialize failed
```

Причины:
- запуск в эмуляторе (Ryujinx/Yuzu — нормально, термодатчиков нет);
- AMS подгрузился с битым ams_mitm — переустановите пак Ряженки
  через `Tools → Установить пак Ряженки`.

То же относится к `psmInitialize` / графику аккумулятора.

## Sysmodule вернулся включённым после перезагрузки

Toggle создаёт/удаляет `<tid>/flags/boot2.flag`. Если файл создался
снова сам — какая-то другая утилита (например ovlSysmodules)
переписывает флаги при старте.

- Проверьте `Tools → Показать лог приложения` на строки
  `sysmodule: ... → disabled` — наша запись должна быть.
- Удалите вручную через Hekate UMS, чтобы убедиться, что AMS читает
  наше состояние.

## Restore завис на стадии «копирование»

Жмёте B → возвращаетесь к Tools. Прерванный restore оставил
`/atmosphere/` в смешанном состоянии. Запустите restore **ещё раз** с
тем же снимком — pre-snapshot будет уже содержать прерванный state
для последующей диагностики, а live /atmosphere восстановится
с нуля.

Если на SD кончилось место — `Tools → Очистка резервных копий`
прежде чем повторять.
